# Retrospective: 2026-05-09 — First Full Dogfooding Day

## What worked

1. **Task-as-brief pattern.** Giving a developer agent a task ID instead of a prose brief worked perfectly when the task description was rich enough. The graph traversal bug fix (a069231d) went from task ID to merged fix in 2 minutes.

2. **Parallel eng-managers with different slugs.** Running eng-store, eng-cli, eng-triage, eng-mcp, eng-infra, eng-docs simultaneously with no slug collisions. Different names = different agents = true parallelism.

3. **Coordinator bookkeeping pattern.** Moving tasks to `working` stage before starting agents gave accurate board state during execution. All 14 remediation tasks had clean stage transitions.

4. **Triage audit.** Dispatching an agent to cross-reference 43 tasks against the codebase found 5 more already-done items and confirmed 37 genuinely needed. Cleaned up the backlog in one pass.

5. **MCP adapter shipped.** 10 tools exposed via `ft mcp serve`, Claude Code config included. From brief to merged in ~40 minutes. This is the product differentiator.

6. **Dogfooding notes as appendable log.** `.design/dogfooding-notes.md` captured observations in real-time with evidence and implications. Became the basis for this retro.

## What didn't work

1. **Agents that can't push lose all work on deletion.** eng-infra and eng-docs completed their work but couldn't push (GitHub App lacks `workflows` permission). When agents were deleted, commits were lost. Three tasks showed `completed` in ft with no code on origin. **Root cause:** No checkpoint mechanism between "work done" and "work delivered."

2. **Permission cascading between parallel agents.** eng-infra's unpushable `.github/workflows/` commit poisoned eng-docs' push path because they shared clone lineage. An unrelated agent was blocked by another agent's permission failure. **Root cause:** Hub clones share git history; one bad commit blocks all subsequent pushes.

3. **`.claude/` directory is a permission trap.** eng-mcp stalled twice trying to write `.claude/mcp.json` — interactive permission prompts that non-interactive agents can't approve. **Root cause:** Claude Code gates sensitive paths with interactive prompts; agents have no way to pre-approve.

4. **Task state drift from failed pushes.** Agents closed tasks as `completed` before confirming push. Coordinator had to manually verify git log and reopen 3 tasks. **Root cause:** "done" means "committed locally" to the agent, but "done" means "on origin/main" to the project.

5. **Stale backlog from document-only decomposition.** TPM agent created 57 work-tasks from the roadmap document without checking git history. 19 (33%) were already implemented. Burned agent-slots verifying completed work. **Root cause:** Backlog creation didn't cross-reference codebase state.

6. **Thin task descriptions.** Remediation tasks had one-line descriptions ("Fix timing-safe token comparison. Size: S.") — not enough for task-as-brief. Had to supplement with "read .design/remediation-plan.md." **Root cause:** TPM optimized for quantity of tasks over quality of descriptions.

## Key insight

> Farm Table is a task runtime for AI agents. The dogfooding revealed that the biggest risks aren't in the task data model — they're in the workflow between "agent says done" and "work is actually delivered." The tool tracks task state, but delivery confirmation (push, deploy, verify) is outside its scope. Bridging that gap — either in the tool or in the orchestration layer — is the highest-leverage improvement.
